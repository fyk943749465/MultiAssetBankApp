# 全站 API 数据来源规则

本文约定本仓库**对外 HTTP API**（含前台、后台、管理端）在实现时，**读**与**写**分别以哪一侧为权威，适用于各业务模块；与链上、子图、PostgreSQL 打交道的接口均应遵循。

## 1. 查询类接口（GET / 只读）：以子图为准

- **原则**：列表、详情、统计、只读管理视图等**不改变链上或库内状态**的接口，在**该读路径已由子图（The Graph / 自建子图）承载**的前提下，**优先以子图为数据源**。尚未建子图的模块，只读接口可暂以 PostgreSQL 等读存储实现；**一旦对应域接入子图，应与本条对齐**，避免长期依赖重 RPC 扫块。
- **目的**：子图面向链上事件已做索引与关联查询，能显著减少 RPC 压力与复杂联表，**保证查询效率与可扩展性**。
- **降级**：子图未配置、请求失败或返回不完整时，可按接口设计**回退到 PostgreSQL**（如索引器写入的缓存表、事件流水表），并在响应或文档中说明数据来源及 `total` 等字段语义（如「窗口内条数」）。

## 2. 状态变更类接口（POST / 提交交易 / 审核动作等）：以 PostgreSQL 为准

- **原则**：会触发**合约调用、交易构造、权限校验、业务状态推进**的接口，**以 PostgreSQL 中已持久化的读模型为准**（与链上最终一致、由索引器或同步任务维护的数据）。
- **目的**：避免仅依据子图延迟或索引滞后视图构造交易，导致**参数与链上真实状态不一致**。**以库内权威快照决策**，降低误操作与失败回滚成本。
- **说明**：子图仍可用于**展示与辅助提示**；**是否允许执行某动作、携带哪些参数 / calldata**，应以库内记录及必要时链上只读校验（如 `eth_call`）为准。

## 3. 小结

| 类型           | 权威数据源     | 主要目标   |
|----------------|----------------|------------|
| 查询 / 只读    | 子图（优先）   | 效率、规模 |
| 写 / 状态变更  | PostgreSQL     | 正确、防错 |

新增或修改接口时，应先归类为「只读查询」或「状态变更」，再按上表选择主数据源；若存在例外，须在接口注释与对外文档中写明。
