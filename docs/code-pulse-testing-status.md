# Code Pulse 众筹模块 — 测试进度说明

> 最后更新：根据当前联调记录整理；**添加 / 移除项目开发者**均已纳入「已测试」。

本文档区分 **已测试** 与 **未测试 / 待测** 能力，便于回归与发版前自检。合约与 UI 行为以实际部署为准。

---

## 一、已测试功能

| 序号 | 能力 | 说明 |
|------|------|------|
| 1 | 管理员设置白名单 | 提案发起人（proposal initiator）白名单相关流程 |
| 2 | 管理员审核提案 | 提案进入审核通过/拒绝等路径 |
| 3 | 提案发起人提交提案 | 创建提案 |
| 4 | 提案发起人提交第一轮次 | 首轮参数提交（如 target / duration 等，以合约为准） |
| 5 | 管理员审核轮次 | 轮次审核通过后可进入发起众筹 |
| 6 | 提案发起人发起众筹 | 上线众筹（CrowdfundingLaunched 等） |
| 7 | 捐款人捐款 | 捐助路径可走通 |
| 8 | **添加项目开发者** | 在活动上添加开发者（链上 DeveloperAdded；后端/子图与前端展示已对齐关注） |
| 9 | **移除项目开发者** | 链上 `removeCampaignDeveloper` / `DeveloperRemoved`；预检以 PostgreSQL `cp_campaign_developers` 为准（须等 RPC 索引同步）；名单与链上一致性已验证 |

---

## 二、未测试 / 待测功能

| 序号 | 能力 | 说明 |
|------|------|------|
| 1 | 众筹失败与退款 | 未达标/失败态下捐助人退款（RefundClaimed 等） |
| 2 | 众筹成功后的分阶段放款 | 里程碑审批、开发者领取份额（`approveMilestone` / `claimMilestoneShare` 等） |
| 3 | 第二轮（及后续轮）众筹 | 多轮次衔接、状态与资金隔离 |
| 4 | 平台资金相关 | 若有平台捐赠/提现等事件，需按合约单独设计用例 |
| 5 | 紧急与运维能力 | 若合约含 `pause` / 所有权转移等，需在暂停前后分别验证 |
| 6 | 边界与异常 | 0 金额、重复操作、越权调用、deadline 前后行为等 |
| 7 | 索引与子图一致性 | RPC 索引、`cp_event_log`、子图与前端「数据来源」展示在关键操作后是否一致 |

---

## 三、建议优先补测顺序（参考）

1. **退款全流程**（失败路径与资金安全）  
2. **里程碑审批 + 领取**（成功路径资金释放）  
3. **第二轮众筹**（与第一轮衔接）  
4. 再补边界、权限与索引/子图对照
